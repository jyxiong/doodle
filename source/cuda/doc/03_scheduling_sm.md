# 三、CUDA 线程调度与 SM

## 3.1 SM（流式多处理器）概述

- SM（Streaming Multiprocessor）是 GPU 的核心执行单元
- **核心职责：** 调度 Warp、分配寄存器与共享内存、执行指令、处理访存请求
- 一个 GPU 包含多个 SM（如 A100 有 108 个 SM）

### A100 SM 驻留能力参考

| 资源 | A100 上限 |
|------|-----------|
| Block / SM | 32 |
| Warp / SM | 64 |
| Thread / SM | 2048 |

> 不同架构（Volta、Ampere 等）的具体数值有所不同。

---

## 3.2 Block 分配到 SM 的流程

```
核函数启动 → 生成 Grid（含所有 Block）
          → 硬件调度器扫描空闲 SM
          → 判断 SM 是否有足够寄存器、共享内存
          → 将 Block 分配到空闲 SM
          → SM 分配资源，划分线程为 Warp
          → Warp 调度器调度 Warp 执行指令
```

### Block 执行与释放机制

- Block 内所有线程执行完毕后，自动释放寄存器、共享内存资源
- SM 无抢占机制，Block 释放后才能接收新 Block

---

## 3.3 Warp 调度机制

- SM 内有多个 **Warp 调度器**，选取就绪的 Warp（无访存阻塞、无同步等待）发射指令
- Warp 遇到访存阻塞或同步指令时，调度器**零开销切换**到其他就绪 Warp

### 访存延迟隐藏（Latency Hiding）

```
Warp A 发出全局内存访存请求（延迟 ~数百 ns）
  → 硬件立即切换到 Warp B 执行计算
  → Warp A 数据就绪后重新进入就绪队列
  → 计算掩盖了访存等待，SM 持续工作
```

**影响 Warp 切换效率的因素：**
1. SM 内驻留的 Warp 数量（越多越好）
2. 访存延迟的长短
3. 指令密度（计算指令越多，越容易掩盖访存延迟）
4. Warp 的就绪比例

---

## 3.4 GPU 调度 vs CPU 调度

| 对比维度 | GPU | CPU |
|----------|-----|-----|
| 调度方式 | 协作式硬件 Warp 调度 | 抢占式软件调度 |
| 核心特点 | 核心轻量，数量多（数千） | 核心复杂（乱序执行、分支预测） |
| 切换开销 | 零开销 Warp 切换 | 较高的上下文切换开销 |
| 优化目标 | 高吞吐量 | 低延迟 |
| 适合任务 | 并行、计算密集 | 串行、控制密集 |

---

## 3.5 动态并行（Dynamic Parallelism）

**定义：** 核函数（父内核）执行过程中，可动态启动子核函数。

**调度规则：** 子内核由父线程启动，共享父内核所在 SM 的硬件资源。

**资源限制：**
- 受父内核剩余寄存器、共享内存限制
- 不能无限嵌套
- 子内核启动有一定开销

---

## 3.6 核函数启动开销

**组成部分：**
- 驱动程序上下文切换
- 命令队列提交
- Block 分配
- 资源分配
- Warp 调度

**降低方法：**
1. 合并小内核为大内核，减少启动次数
2. 使用多 Stream 并发执行，掩盖启动开销
3. 启用核函数缓存
4. 减少驱动程序调用次数

---

## 3.7 SM 理论占用率（Occupancy）

$$\text{理论占用率} = \frac{\text{SM 内活跃 Warp 数}}{\text{SM 最大支持 Warp 数}} \times 100\%$$

**影响占用率的核心资源：**
- 单 Block 线程数
- Block 数量
- SM 的寄存器总量
- SM 的共享内存容量
- SM 的 Warp 最大驻留数

**提升占用率的方法：**
1. 降低单个 Block 的寄存器和共享内存消耗，让 SM 能驻留更多 Block
2. 将 Block 内线程数设为 **32 的整数倍**，提升 Warp 利用率
3. 调整 Grid 大小，保证 SM 有足够的 Block 可分配
4. 优化编译器选项，减少寄存器占用
